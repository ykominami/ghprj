# diff_update 外部仕様

## 1. 目的

`ghrepo list` 実行時の最新DB更新条件を、**レコード単位**で明確化する。  
最新DBは、最新スナップショットDBと比較して、**各レコードごとに `count` 以外のフィールド差分が1つ以上ある場合のみ当該レコードを更新**する。

## 2. 適用範囲

- 対象コマンド: `ghrepo list`
- 対象データ:
  - 最新DB（`db`）
  - 最新スナップショットDB（`repolist/{count}/db.yaml`）
- 非対象コマンド: `ghrepo fix`（本仕様の直接対象外）

## 3. 用語定義

- **最新DB**: 運用上の最新版として参照する `db`。
- **最新スナップショットDB**: 直近取得回に対応する `repolist/{count}/db.yaml`。
- **比較対象フィールド**: Repo レコード内の `count` を除く全フィールド。
- **更新対象レコード**: 同一 Repo キーで比較した結果、比較対象フィールドに差分があるレコード。
- **新規レコード**: 最新DBには存在せず、最新スナップショットDBにのみ存在するレコード。
- **欠落レコード**: 最新DBには存在し、最新スナップショットDBに存在しないレコード。

## 4. 処理仕様

1. GitHub API 取得結果から今回分データを作成する（管理項目 `count` を含む）。
2. 今回分データを最新スナップショットDBとして保存する。
3. 最新DBと最新スナップショットDBを Repo キー単位で比較する。
   - 比較時は `count` を除外する。
4. 比較結果をレコード単位で反映する。
   - 更新対象レコード: 最新DBの該当レコードを置換更新する。
   - 新規レコード: 最新DBへ追加する。
   - 欠落レコード: 最新DBから削除しない（据え置く）。

## 5. 判定ルール

### 5.1 既存キー（両DBに存在）

- `count` 以外に差分あり: 当該レコードを更新する。
- 差分が `count` のみ: 当該レコードは更新しない。
- 完全一致: 当該レコードは更新しない。

### 5.2 新規キー（スナップショットのみ存在）

- 当該レコードを最新DBへ追加する。

### 5.3 欠落キー（最新DBのみ存在）

- 当該レコードは削除しない。
- 既存値をそのまま保持する。

## 6. 例外・フォールバック

- 最新DBが存在しない場合は、初回作成としてスナップショット内容を最新DBへ出力する。
- 比較対象ファイルの読み込み失敗・形式不正時は、安全側としてスナップショット内容を最新DBへ出力する。
- 例外処理を適用した場合は警告ログを出力する。

## 7. 受け入れ条件

### ケース1: 同一キーで `count` のみ差分

- 期待結果: 当該レコードは更新されない。

### ケース2: 同一キーで通常フィールド差分（例: `description` 変更）

- 期待結果: 当該レコードのみ更新される。

### ケース3: 新規キーあり

- 期待結果: 新規レコードが追加される。

### ケース4: 欠落キーあり（スナップショット側に存在しない）

- 期待結果: 欠落レコードは削除されず保持される。

### ケース5: `count` 以外を含め完全一致

- 期待結果: どのレコードも更新されない。

### ケース6: 最新DB欠落または比較失敗

- 期待結果: スナップショット内容が最新DBへ出力される（初回作成/フォールバック）。

## 8. 互換性

- 既存のスナップショット保存（`repolist/{count}/db.yaml`）は維持する。
- `fetch` 更新ロジックの仕様は変更しない。
- 本仕様で変更されるのは、最新DBへの**反映単位と判定条件**のみ。
